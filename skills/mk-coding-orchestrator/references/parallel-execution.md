# Parallel Execution

Механика параллельного запуска агентов. Применяется когда задачи **не мешают друг другу**. Цель — скорость без гонок за общее git-состояние.

Главный принцип: **агенты работают каждый в своей копии проекта и слепы к git. Ветками и слиянием владеет только оркестратор.**

---

## Что можно параллелить, а что нет

| Что | Можно ли | Почему |
|---|---|---|
| Спеки разных независимых этапов | ✅ всегда | отдельные файлы, append-only, никакого общего git-состояния |
| Планы разных независимых этапов | ✅ всегда | то же самое |
| Задачи **внутри** одного плана (Фаза C) | ✅ если `touches` не пересекаются | требует изоляции (см. ниже) |
| Фазы A/B/C одного этапа между собой | ❌ никогда | план зависит от спеки, код от плана — жёсткая очередь |

Если задачи делят файлы или одна зависит от другой — **независимости нет**, идут последовательно. Это не сбой, а честная деградация: лучше медленно и верно, чем гонка.

---

## Манифест параллелизма (выдаёт писатель плана, Фаза B)

Писатель плана дополнительно описывает, какие задачи можно делать одновременно. Формат — секция в конце плана:

```markdown
## Манифест параллелизма

- id: T1
  depends_on: []
  touches: ["src/api/**"]
- id: T2
  depends_on: []
  touches: ["src/ui/**"]
- id: T3
  depends_on: [T1]
  touches: ["src/api/routes.ts", "src/api/handlers.ts"]
```

- **id** — короткий идентификатор задачи плана
- **depends_on** — id задач, которые должны быть готовы раньше (пусто = ни от чего не зависит)
- **touches** — файлы/папки, которые задача будет менять (glob-маски)

**Правило для писателя плана:** `touches` — это обещание. Если две задачи без взаимной зависимости трогают непересекающиеся файлы — их можно гнать параллельно. Если сомневаешься, трогает ли задача чужой файл — впиши его в `touches` (лучше перестраховаться: пересечение просто переведёт задачи в последовательный режим).

Если манифеста в плане нет или все задачи пересекаются — **Фаза C идёт последовательно**, как раньше (один реализатор за раз).

---

## Построение волн

Оркестратор читает манифест и раскладывает задачи на волны:

1. Взять все задачи, у которых все `depends_on` уже готовы (на старте — с пустым `depends_on`).
2. Из них в одну волну попадают только те, чьи `touches` **попарно не пересекаются**.
3. Если две готовые задачи пересекаются по `touches` — в эту волну берётся одна, вторая ждёт следующей волны.
4. Ширина волны ограничена капом (по умолчанию **3 агента**, настраивается в промпте запуска).
5. После завершения волны — пересчитать: какие задачи теперь разблокировались.

Пример выше → волна 1: `[T1, T2]` параллельно; волна 2: `[T3]` (ждала T1).

---

## Изоляция: своя копия проекта на агента

Каждый параллельный реализатор запускается в **отдельном worktree** — это отдельная физическая папка с той же кодовой базой на своей ветке. Через инструмент `Agent`:

```
isolation: "worktree"
```

Харнесс сам создаёт worktree до старта агента и убирает после. Агент **не создаёт его руками**.

### Контракт «git-слепого» агента

В промпт каждого параллельного реализатора жёстко вписать:

```
Твоя рабочая директория: [абсолютный путь к worktree].
Работай ТОЛЬКО в ней.

ЗАПРЕЩЕНО:
- git checkout, git switch, git branch, git worktree, git merge, git rebase
- cd за пределы рабочей директории
- любые правки файлов вне твоего списка задач

РАЗРЕШЕНО:
- править файлы из твоего задания
- один git commit в конце работы (git add -A && git commit)
```

Почему так: переключение веток в общем дереве — ровно тот сценарий, где агенты начинают видеть чужие изменения как свои, паниковать и откатывать. Своя папка + запрет на ветко-команды делает это физически невозможным.

## Fan-in: сборка результатов (делает только оркестратор)

Когда все агенты волны закончили, оркестратор по очереди (не параллельно) забирает их работу:

1. **Проверка полосы.** Для каждого агента: `git diff --name-only` его ветки против базы. Все ли изменённые файлы попадают в заявленный `touches`?
   - Да → принять.
   - Нет (полез за свою полосу) → **отбросить дифф этого агента целиком и эскалировать пользователю**. Не давать агенту «чинить самому» — самостоятельный откат и есть источник хаоса.
2. **Слияние.** Влить принятые ветки в основную по очереди. Раз `touches` непересекающиеся — конфликтов быть не должно. Если конфликт всё же возник — значит, `touches` солгал: остановиться и эскалировать, не разрешать конфликт вслепую.
3. **Коммит точки слияния** оркестратором: `git commit -m "merge(impl): волна N — задачи T1, T2"`.

---

## Контракт коммитов при параллелизме

Базовое правило проекта — «коммит после каждого агента». При fan-out оно уточняется:

- **Внутри волны:** каждый агент коммитит у себя в worktree (один коммит в конце своей работы).
- **После fan-in:** оркестратор делает отдельный коммит точки слияния на основной ветке.

То есть на волну из N агентов приходится N коммитов в worktree + 1 коммит слияния. Гейт (Фаза D, ревью кода) запускается **один** на весь объединённый дифф этапа, а не на каждую задачу.

## Отчётность пользователю

Оркестратор рапортует о параллельной работе **простым языком**, без жаргона worktree/fan-in:

```
Запускаю 2 задачи параллельно:
  • T1 — API (трогает src/api/)
  • T2 — интерфейс (трогает src/ui/)
Каждая в своей изолированной копии, между собой не пересекаются.
```

После сборки:

```
Обе задачи готовы и слиты. T3 зависела от T1 — запускаю её следующей.
```

Это учёт состояния конвейера (исключение 2 контракта), а не написание содержимого — контракт оркестратора не нарушается.

---

## Когда НЕ параллелить (сводка)

- Манифест отсутствует → последовательно.
- Все задачи пересекаются по `touches` → последовательно.
- `Agent` не поддерживает `isolation: "worktree"` в текущем харнессе → последовательно + сообщить пользователю (честная деградация, как с CLI в `spawn-recipes.md`).
- Любое сомнение в независимости → последовательно. Скорость не стоит сломанного диффа.
