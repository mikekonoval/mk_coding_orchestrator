# Roadmap Orchestrator

## Жёсткий контракт оркестратора

**Оркестратор не пишет содержимое артефактов.** Спека, план и код создаются только заспавненными агентами.

Два исключения, которые остаются в основной сессии:
1. **Диалог с пользователем** — уточняющие вопросы, сбор решений, подтверждение этапа
2. **Учёт состояния конвейера** — отметки в роадмапе (`- [x]`), статусные заметки, чек-лист дымового теста

Если пользователь просит «просто напиши спеку сам» — заспавни писателя и передай ему, не пиши инлайн.

---

## Шаг 1. Приём роадмапа

Найти файл роадмапа:
1. Проверить `ROADMAP.md` в корне
2. Проверить `docs/roadmap*.md` или `docs/ROADMAP.md`
3. Если не найден — спросить пользователя путь или предложить создать

Формат роадмапа — чек-лист этапов:
```markdown
## Этапы проекта
- [ ] Этап 1: описание
- [x] Этап 2: описание (завершён)
- [ ] Этап 3: описание
```

Выбрать **следующий незавершённый** этап (`- [ ]`). Подтвердить выбор с пользователем перед стартом.

Это же правило задаёт **resume** в новой сессии: открыть роадмап, найти первый незакрытый пункт — продолжить с него.

---

## Шаг 2. Конвейер этапа

Каждый этап проходит четыре фазы в указанном порядке. Следующая фаза начинается только когда предыдущая прошла гейт ревью.

### Фаза A. Спека

1. Оркестратор ведёт диалог с пользователем: цели этапа, ограничения, решения по дизайну. Резюмировать решения списком.
2. Заспавнить агента-писателя:
   - Передать: резюме решений, этап из роадмапа, абсолютный путь к `superpowers:brainstorming` SKILL.md
   - Инструкция: прочитать SKILL.md brainstorming перед написанием, создать документ спеки в `docs/superpowers/specs/`
3. **Гейт**: кросс-модельная волна ревью спеки (см. Шаг 3)

### Фаза B. План

1. Заспавнить агента-писателя плана:
   - Передать: путь к спеке, этап из роадмапа, абсолютный путь к `superpowers:writing-plans` SKILL.md
   - Инструкция: прочитать SKILL.md writing-plans перед написанием, создать план в `docs/superpowers/plans/`
2. **Гейт**: кросс-модельная волна ревью плана (см. Шаг 3)

### Фаза C. Реализация

1. Заспавнить агента-реализатора:
   - Передать: путь к плану, абсолютный путь к `superpowers:subagent-driven-development` SKILL.md
   - Инструкция: прочитать SKILL.md перед реализацией
   - Выбрать модель по рубрике → `references/model-routing.md`
2. Внутренние per-task ревью superpowers **не считаются** волнами по R4 — они остаются внутри фазы C

### Фаза D. Ревью кода

1. **Гейт**: кросс-модельная волна ревью кода (см. Шаг 3)
   - Диапазон ревью: весь дифф этапа (BASE..HEAD по всем задачам фазы C)

После прохода гейта D — отметить этап завершённым в роадмапе (`- [x]`), перейти к следующему.

---

## Шаг 3. Цикл волны кросс-модельного ревью

Полная механика — в `references/review-protocol.md`.

Кратко:
1. Выбрать ревьюера: **другая модель/CLI, чем писатель** (матрица → `references/spawn-recipes.md`)
2. Построить нейтральный промпт по шаблону из `references/prompts/` (spec/plan/code)
3. Заспавнить ревьюера
4. Разобрать находки по контракту (Critical/Important/Minor)
5. Если Critical или Important есть:
   - Заспавнить фиксера с находками дословно
   - Счётчик волн < кап (по умолчанию 5)? → вернуться к п.2
   - Счётчик ≥ кап → эскалировать к пользователю
6. Если только Minor или пусто → **гейт пройден**

---

## Шаг 4. Завершение этапа

- Отметить этап в `ROADMAP.md`: `- [ ]` → `- [x]`
- Предложить пользователю перейти к следующему этапу или остановиться
